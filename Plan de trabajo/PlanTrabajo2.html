4. Secuencia de Ejecución

A.1 (radio) → limpiar imports → validar RAM ↓ A.2 (config_system) → limpiar imports → validar RAM ↓ A.3 (config_fase3) → limpiar imports → validar RAM ↓ A.4 (email_state) → limpiar imports → validar RAM ↓ B.1 (emails→alertas) → limpiar imports → validar ↓ C.1 → C.2 → C.3 (ITV+PRG) → limpiar imports → validar ↓ D.1 → D.3 (zona horaria+horas estado) → limpiar imports → validar ↓ F (fase3_utils) → limpiar imports → validar ↓ G final (revisión global de imports muertos)

6. Fase A — Fusiones de Módulos Pequeños Riesgo Mínimo

🆕 Orden de ejecución corregido: A.1 → A.2 → A.3 → A.4

Razón: A.1 es independiente del resto y sirve para calibrar el proceso de medición de RAM. A.2 crea config_system.py. A.3 añade ConfigFase3 a ese módulo ya existente. A.4 fusiona email_state.py → alertas.py, que ya puede importar config_system.py si necesita leer config.

Checkpoint RAM obligatorio tras cada sub-fase. Ver §15 (Parte 4).

A.1: Fusionar radio_base.py → radio_manager.py PRIMERO

A1
radio_base.py + radio_manager.py

Motivo: radio_base.py es una clase de 6 atributos sin métodos. radio_manager.py es su único usuario. Es completamente independiente de las demás fusiones.

Acción: Mover la clase RadioBase al inicio de radio_manager.py, eliminar from radio_base import RadioBase, borrar radio_base.py.

# En radio_manager.py, reemplazar el import: # from radio_base import RadioBase ← ELIMINAR class RadioBase: def __init__(self): self.crc_actual = False self.rx_iq_actual = False self.frecuencia_actual = None self.sf_actual = None self.bw_actual = None self.cr_actual = None self.syncword_actual = None self.preamble_actual = None

Impacto: -1 fichero, -1 import en RAM. Validación: Inicializar radio en fase3.

🆕 Ventaja de empezar por A.1: Al ser independiente, si algo falla en la medición de RAM o en el proceso de fusión, no afecta a ninguna otra fase. Sirve como "prueba de concepto" para el resto del refactor.

A.2: Fusionar estado.py + configuracion.py → config_system.py SEGUNDO

A2
estado.py + configuracion.py → config_system.py

Motivo: Ambos gestionan persistencia de configuración/estado en JSON. Son 60 + 50 líneas, muy pequeños para justificar módulos separados.

⚠️ Cuidado con el circular: Si configuracion.py o estado.py importan actualmente logger.py (para log_error), la fusión crearía config_system.py ↔ logger.py.

Solución: config_system.py NO importa logger.py. Usa raise en lugar de log_error. El llamador captura y loguea.

Acción: Crear config_system.py con todo el contenido de ambos, eliminando imports de logger.py. Actualizar imports en todos los ficheros que los usaban.

Impacto: -2 ficheros, -1 import por fichero que usaba ambos. Validación: Arranque completo por todas las fases.

A.3: Fusionar config_fase3.py → config_system.py TERCERO

A3
config_fase3.py → config_system.py

Motivo: ConfigFase3 (100 líneas) es una cache de config específica para fase3. Hace parsing de horas, sweep, ventilador, heartbeat — todo es configuración. No justifica módulo propio cuando config_system.py ya gestiona config global + estado de fase.

Acción: Mover la clase ConfigFase3 al final de config_system.py. Cambiar import en fase3.py de from config_fase3 import ConfigFase3 a from config_system import ConfigFase3.

# config_system.py (al final, tras funciones de estado) class ConfigFase3: """Cachea parámetros de config relevantes para fase3.""" def __init__(self, config): ... def _parsear_horas(self, horas_raw): ...

Impacto: -1 fichero. fase3.py sigue funcionando igual. config_system.py se convierte en el único punto de entrada para toda la configuración del sistema.

Validación: Arrancar fase3, verificar que ConfigFase3 lee correctamente horas fijas, sweep, ventilador.

A.3b: Evaluar sweep_params.py OPCIONAL

A3b
¿Fusionar sweep_params.py → config_system.py?

Motivo: sweep_params.py (80 líneas) gestiona parámetros de sweep de frecuencia. Si es solo un diccionario de constantes, justifica fusión con config_system.py. Si contiene lógica de cálculo de frecuencias Doppler o algoritmos de búsqueda, debe quedar independiente.

⚠️ Decisión condicional:
  • SÍ fusionar si sweep_params.py contiene solo constantes (dicts de frecuencias, rangos, pasos).
  • NO fusionar si contiene funciones de cálculo (ej. calcular_frecuencia_doppler(), ajustar_rango_sweep()). En ese caso, renombrar a sweep_engine.py para claridad.

Acción: Revisar contenido de sweep_params.py. Si es solo datos, mover a config_system.py como SWEEP_DEFAULTS o similar. Si tiene lógica, dejar como módulo hoja y renombrar.

Impacto: -1 fichero si se fusiona, o +0 con nombre más claro. Validación: Sweep funciona correctamente en fase3.

A.4: Fusionar email_state.py → alertas.py CUARTO

A4
email_state.py → alertas.py

Motivo: email_state.py gestiona horas fijas de envío (cuándo). alertas.py gestiona el envío SMTP (cómo). Ambos son "email".

🆕 Ventaja del orden corregido: Como config_system.py ya existe (A.2), alertas.py puede importarlo limpiamente si necesita leer config para las horas de estado. No hay conflicto con D.3.

Acción: Mover la clase EstadoEmail al final de alertas.py (antes del if __name__ == "__main__"). Cambiar imports en fase3.py de from email_state import EstadoEmail a from alertas import EstadoEmail.

Impacto: -1 fichero (28 → 22 tras A.1-A.4 + F; o 21 si A.3b se ejecuta). Validación: Enviar email de estado en hora fija.
