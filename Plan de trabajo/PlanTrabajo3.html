9. Fase C — Reorganizar itv_manager.py Riesgo Medio

C.1: Mover preparación de datos ITV de fase4.py → itv_manager.py; envío a alertas.py

C1
Datos ITV → itv_manager.py; Construcción+Envío ITV → alertas.py

Motivo: fase4.py actualmente construye manualmente el cuerpo del email ITV (140 líneas) leyendo JSON de itv_manager.py. Esto dispersa la lógica de email en tres sitios: alertas.py (SMTP), fase4.py (construcción ITV), itv_manager.py (preparación JSON). La intención de B.1 es centralizar todo lo relacionado con emails en alertas.py.

Acción:

  1. itv_manager.py: Mantiene _preparar_email_itv() (ya existe). Añade método obtener_datos_email_itv() que devuelve dict con métricas, motivos, checklist, acciones, días — listo para consumir.
  2. alertas.py: Añade construir_email_itv(datos_itv) que recibe el dict y construye el cuerpo del email (las 140 líneas actuales de fase4.py). Añade enviar_email_itv(datos_itv, debug_activo) que construye + envía vía enviar_correo_bloques().
  3. fase4.py: Solo orquesta: if itv.hay_email_itv_pendiente(): alertas.enviar_email_itv(itv.obtener_datos_email_itv(), DEBUG_MODO)

Impacto: fase4.py pasa de ~420 a ~280 líneas. alertas.py centraliza todo lo de email (estado, capturas, ITV, reporte diario). itv_manager.py se enfoca en métricas y triggers.

Principio: alertas.py = fábrica de emails + envío SMTP. itv_manager.py = lógica de negocio ITV (métricas, triggers, reset). fase4.py = orquestador puro.

C.2: Unificar detección PRG en placa.py

C2
PRG unificado → placa.py

Motivo: La misma lógica de 3 pulsaciones en 5 segundos está duplicada en fase3.py y fase4.py. placa.py ya tiene prg_pulsado() y led_blink().

Acción: Añadir a placa.py:

def detectar_pulsacion_prg(pulsaciones_objetivo=3, ventana_ms=5000, max_duracion_ms=1000): """Detecta N pulsaciones cortas de PRG dentro de una ventana de tiempo.""" if not prg_pulsado(): return False pulsaciones = 1 t_inicio = time.ticks_ms() while prg_pulsado(): time.sleep_ms(50) if time.ticks_diff(time.ticks_ms(), t_inicio) > max_duracion_ms: return False while time.ticks_diff(time.ticks_ms(), t_inicio) < ventana_ms: if prg_pulsado(): t_pulso = time.ticks_ms() while prg_pulsado(): time.sleep_ms(50) if time.ticks_diff(time.ticks_ms(), t_pulso) > max_duracion_ms: pulsaciones = 0 continue pulsaciones += 1 if pulsaciones >= pulsaciones_objetivo: return True time.sleep_ms(50) return False

Actualizar:

  • fase3.py: reemplazar _comprobar_prg() por if placa.detectar_pulsacion_prg(): ...
  • fase4.py: reemplazar _detectar_confirmacion_itv_prg() por if placa.detectar_pulsacion_prg(): ...

Impacto: -80 líneas duplicadas. Validación: Probar PRG en fase3 y fase4.

C.3: Mover forzar transición fase4 desde itv_manager.py CORREGIDO

C3
forzar_itv() → fase3.py (orquestador)
Plan original (incorrecto): Mover guardar_fase(4) a fase4.py. Esto crearía un circular: fase3.py → itv_manager.py → fase4.py → ... (fase4 probablemente importa fase3 o itv_manager).
Solución corregida: itv_manager.py no sabe de fases. Solo marca itv_pendiente = True y devuelve un flag. fase3.py (el orquestador) lee el flag y ejecuta guardar_fase(4).

Arquitectura corregida:

# itv_manager.py — solo lógica de negocio, sin imports de fases class ITVManager: def __init__(self): self.itv_pendiente = False ... def evaluar_trigger(self): """Evalúa si se debe forzar ITV. Devuelve True si hay que transicionar.""" if self.dias_sin_itv >= self.umbral_dias: self.itv_pendiente = True self._crear_estado_pendiente_json() # su responsabilidad: guardar su estado return True return False def _crear_estado_pendiente_json(self): """Guarda estado_pendiente.json para que fase4 lo lea.""" # No llama guardar_fase(4) — eso es orquestación ... # fase3.py — orquestador, decide transiciones from config_system import guardar_fase from itv_manager import ITVManager itv = ITVManager() # ... en el loop principal ... if placa.detectar_pulsacion_prg() or itv.evaluar_trigger(): guardar_fase(4) return # fase3 termina, main.py arranca fase4

Beneficios de esta arquitectura:

  • itv_manager.py no importa config_system.py para guardar_fase (evita circular)
  • itv_manager.py no importa fase4.py (evita circular grave)
  • fase3.py mantiene su rol de orquestador: decide cuándo transicionar
  • Testeable: se puede testear itv_manager.evaluar_trigger() sin cargar todo el sistema de fases

Impacto: itv_manager.py pierde la responsabilidad de orquestación. fase3.py gana una línea de transición. Sin circular.

Punto de inserción en fase3.py

# En fase3.py, en el loop principal, donde actualmente se evalúa PRG: # (sustituye la lógica duplicada _comprobar_prg() por placa.detectar_pulsacion_prg()) if placa.detectar_pulsacion_prg() or itv.evaluar_trigger(): guardar_fase(4) return # fase3 termina, main.py arranca fase4

Nota: Este check se ejecuta en cada iteración del loop principal de fase3, justo después de procesar un paquete RX o durante el sleep entre pases. No debe ir dentro del procesamiento de un pase activo para evitar interrumpir una recepción en curso.

Motivo: obtener_desfase_espana() es lógica pura de zona horaria (DST europeo). No tiene nada que ver con satélites. Es la causa del import circular.

D.2: Aislar N2YO como clase interna en datos_satelites.py POSPUESTO

D2
N2YO como clase interna — NO nuevo módulo

Motivo original: descargar_agenda_completa() es 250 líneas de HTTP + SSL + parsing + serialización. Si N2YO cambia la API, tocas datos_satelites.py que también calcula horas de estado y resuelve parámetros LoRa.

Revisión: En MicroPython cada .py importado consume RAM. El plan A reduce de 28 a ~24 módulos. Crear n2yo_client.py devolvería a ~25. Si la API de N2YO es estable (lo es desde 2018), el beneficio es marginal frente al overhead de import.

Acción revisada: NO crear nuevo módulo. Opciones alternativas (evaluar post-A):

  1. Clase interna N2YOClient dentro de datos_satelites.py — delimita la lógica HTTP sin nuevo fichero.
  2. Función con prefijo _n2yo_ en datos_satelites.py — marca claramente el código de API externa.
  3. No hacer nada — si la API de N2YO sigue estable, el código funciona. Documentar con comentario # API N2YO - si cambia, solo tocar esta función.

Veredicto: POSPUESTO — aplicar opción 2 (prefijo _n2yo_) solo si se toca la función por otro motivo. No justifica refactorización dedicada.
